# System Architecture & Development Learnings

## Phase 2: Backend API Normalization Pipeline (Completed Mar 2026)

During the integration of independent brokerage services into a single unified JSON schema, several high-level architectural patterns were strictly enforced to maintain stability and prevent repeating authentication cycles:

### 1. Unified Broker Service Aggregation
Rather than configuring specific endpoints for different geographic markets, **dynamic instantiation loops** act as the most robust defense. (e.g., Moomoo OpenD contexts natively reject requests outside explicit `TrdMarket` filters. Iterating across *all permutations* guarantees data capture regardless of the end-user's hidden environment variables).

### 2. Defeating Provider Security Restrictions Natively
Third-party APIs (specifically `questrade-api` utilizing `urllib`) frequently succumb to Web Application Firewalls (WAF) blocking requests without `Mozilla/5.0` User-Agent payloads. 
- *Architectural Standard*: Monkey-patch protocol request engines seamlessly within the host process at the global scope level rather than locally rewriting massive third-party library internals.

### 3. Asynchronous Cache Persistence
Caching must prioritize disk or active states (`~/.questrade.json` and in-memory TTL caching) natively to survive Fast-API hot reloads (`uvicorn`). When swapping configuration vectors (like `.env` refresh tokens), the application loader must **actively burst the memory overlay** (`load_dotenv(override=True)`) to ensure the child processes aren't stubbornly carrying stale payloads.

### 4. Zero-Trust API Testing (OAuth Tokens)
Never run test verification scripts utilizing active production tokens dynamically generated by the user if the authorization structure employs a `grant_type=refresh_token` **Single-Use Payload**. Doing so consumes the sequence and locks out the main application. Mimic the output via independent test tokens or purely trace execution paths natively instead.
